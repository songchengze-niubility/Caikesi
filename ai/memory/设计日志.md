## 项目概述

- **类型**：微信小游戏，Cocos Creator 3.8.8。
- **当前方向**：竖屏整体、**核心战斗横向展开**的多人小队自动战斗（参考「向僵尸开炮」自动战斗 + 永恒蔚蓝星球视角）。
- **小队**：3 人——坦克（近战，前）/ 输出（远程）/ 治疗（后），单排一字阵。
- **当前阶段**：战斗框架 + 关卡系统 + 胜利按配置掉装备 + 结算页/关卡推进 + 装备属性接战斗 + 占位美术。**未做**：技能、Buff、真实美术。
- **原始骨架**：项目最初是「放置挂机刷装备」demo（GameManager/DataService/EventCenter 还在），后转向做战斗。

## 玩法定位决策

- **(2026-06-26) 整体竖屏，但核心战斗横向展开**：小队在左、敌人从右进场向左推，战斗集中在屏幕中部横向带。理由：竖屏放置/RPG 的主流战斗布局，上下留白给 UI。
- **(2026-06-26) 单排一字阵，无碰撞**：双方各排成一条线（y=0），纯靠前后（x）分站位。怪无碰撞、可叠在一起群殴。理由：用户明确要这种「站位分前后、不要分行、可叠」的简单阵型。
- **(2026-06-26) 固定站位 + 自动开火**（非拖动操作）、**固定波次打完即胜**：第一版先验证战斗循环与数值，走位后置。
- **职业分工靠站位**：坦克近战冲到最前当肉墙挡住怪，怪贴脸打坦克；输出远程躲后排集火；治疗奶血量最低的队友。形成「扛—奶—输出」循环。
- **(2026-06-30) 当前阶段切到单人近战 dps 验收**：`battle.xlsx` 增加 `Misc.roster=dps`，当前只生成 dps；tank/healer 的 Stats/Classes/资源仍保留。理由：用户要先不用 3 人小队、只做 1 人 dps；用 roster 覆盖能快速切当前玩法形态，同时不破坏装备/美术/测试里已经存在的多职业基础，后续要回到多人只改表。
- **(2026-07-05) 多人小队上阵系统落地：roster 从配置常量变玩家选择**：从"单人近战 dps 验收"恢复多人小队，但玩家选人而非纯自动分配。关键决策：① 角色池=现有固定职业角色（tank/dps/healer，沿用 class=character 1:1，不做英雄收集/同职业多英雄）；② 队伍上限 `squadCap` 做成配置值（现 2 < 角色数 3，**每场必须取舍**——上 2 下 1），以后加职业改配置即可；③ 站位=一字阵前后顺序（沿用「分前后不分行」既定约定，不做自由走位）；④ 改 roster 仅战前（和穿脱装备重开战斗一致）；⑤ 默认出战 `tank+dps`（`Misc.roster` 从 `dps` 翻回）。实现：`squad/SquadModel.ts` 纯逻辑（存 `PlayerData.squad`，非法存档自愈）+ `BattleManager` 构造函数加 `roster` 入参（缺省回退配置）+ `BattleEntry` 上阵面板（占位，挂「英雄」热区）。**平衡取舍**：连重调一起做——`pacing-sim` 按组合参数化，以默认 tank+dps 为基线重跑；双人后仅两台阶失守（L4 稳），压 L7 怪 hp ×1.15、L10 Boss 血 15100→36000 即复绿，不追求所有组合完美平衡；**tank+healer 无输出角色满装也 0%，定为设计取舍（必须带输出），不硬调**；`sim` RUNS=5 有 RNG 抖动，边界门槛调到极值留余量、复跑确认。设计/计划见 `docs/superpowers/{specs,plans}/2026-07-05-squad-deploy*`。

## 战斗框架决策

- **(2026-06-26) 逻辑与渲染分离**：`BattleManager` 纯数据逻辑，`BattleEntry`/`Background` 负责画。换美术只动渲染层，玩法数值不受影响——这是「色块占位」策略的基础。
- **(2026-06-26) 减法伤害公式 + 保底**：`伤害 = max(攻-防, 攻×minDamageRate)`。保底防「防≥攻」完全免伤。所有伤害走 `CombatFormula.calcDamage` 唯一入口，便于以后换公式。
- **(2026-06-26) 战斗维度**：暴击（率+伤害加成）、格挡（率+减伤比例）、闪避（完全免伤）、攻速（间隔=基础/攻速）、伤害加成（攻）、伤害减免（防）。结算顺序：闪避→减法→暴击→格挡→加成→减免。
- **(2026-06-26) 暴击伤害用「加成式」(critDmg) 而非倍率**：`暴伤=基础×(1+critDmg)`。理由：装备/Buff 的「+30% 暴伤」可累加，倍率式不好叠。**为掉装备铺路**。
- **(2026-06-26) 统一属性表 `stats`**：所有战斗属性收进一张 `CombatStats` 表，角色和怪共用；单位持 `stats` 引用（实时调参生效）。加新维度只动 `CombatStats`+表+公式。
- **(2026-06-26) 射程 `range` 并入 stats 表**（原在 classes 行为里）。理由：让「+射程」装备词条和其他属性走同一叠算路径。**为掉装备铺路**。
- **近战移动**：坦克冲向最前的怪贴脸再劈，清场退守。**硬保险**：前压位置 ≤「怪线−射程」，不管怎么调参坦克都不会越到怪前面（曾出过「怪跑到坦克后面」的 bug）。
- **怪被秒在天边的修复**：远程射程从无限改有限（dps range 320），并提高怪血量，保证怪能挺进到坦克面前再被集火。
- **(2026-07-04) 自动技能系统第一版：全自动释放 + 只做伤害类**：战斗保持全自动，技能不手动放——`timer`（冷却循环）和 `attackCount`（攒普攻次数，挥出即计数不管命中）两种触发器；触发就绪但无目标时保留待放。技能运行态（`combat/SkillRuntime.ts` 纯逻辑）挂在**单位实例**上而非职业抽象，后续 Boss 技能直接复用不返工；`BattleManager` 只在两个钩子点（每帧 tick / 普攻挥出）调用。**伤害 = calcDamage × dmgMult**，不开新公式路径（可暴击可闪避，与普攻行为一致）。只做伤害类、不做 Buff/Debuff 容器：Buff 尚无消费场景，为不存在的需求搭 effect 框架违反 YAGNI。3 个技能按钮定位为**状态展示**（进度遮罩/释放闪光，不可点），为将来可能的手动模式留位。数值取舍：技能是节奏点缀（AoE 清波爽感），台阶门槛主要由关卡 hp 承担——技能接入后三台阶失守，压技能倍率 + L7/L10 怪血 ×1.4 后 `sim:pacing` 13 门槛复绿。边界修正：同帧多技能连放时，前序 AoE 清场导致后序技能目标全灭 → 不发空 `skillCast` 事件（触发已重置，损耗可接受）。设计文档见 `docs/superpowers/specs/2026-07-04-auto-skill-system-design.md`。

## 关卡系统

- **(2026-06-26) 怪物类型表 `enemyTypes` + 关卡表 `levels` 全配置化**：
  - 怪类型（丧尸/疾行者/重装）各有属性、速度、体型、攻击间隔、颜色。
  - 关卡每关多波，每波可**混合多种怪**（类型+数量+间隔+可选 hp 覆盖）。
  - 打完当前关所有波 = 通关。`startLevel` 选默认进哪关。

- **(2026-06-28) 关卡进度独立成 `ProgressModel`，胜利结算页驱动下一关**：`ProgressModel` 只管当前关、最高解锁和通关后解锁下一关；`BattleEntry` 负责在胜利瞬间发奖励、保存进度并显示结算页（下一关/背包/重打）。理由：关卡进度不是装备存储，也不应埋进 `BattleManager`；`BattleManager` 只判断本局胜负，跨局推进由入口层串联。

- **(2026-06-28) 多系统保存共用 `PlayerDataStore` 缓存**：装备和进度都会写 `PlayerData`，因此抽出 `core/data/PlayerDataStore.ts` 作为共享 load/save 缓存。理由：如果 `InventoryPersistence` 和 `ProgressPersistence` 各自缓存一份 PlayerData，后保存的一方可能用旧快照覆盖另一方字段；共享缓存让本地存档在 MVP 阶段保持简单且不互相踩。

- **(2026-07-01) 服务层先对齐，但不重写现有闭环**：新增 seed `Random`、统一 `RewardBundle`、`LootService`、`InventoryService`、`EquipmentService`，`BattleEntry` 胜利奖励改走 `LootService`，但仍复用现有 5 部位/5 品质、`drop.xlsx` 和背包模型。理由：用户要按策划案逐步落地，第一步应建立可复用接口和可验收边界，而不是把已跑通的战斗/装备闭环推倒重写。

- **(2026-07-01) 宝箱来源以在线击杀/关底为准，离线只模拟同规则**：宝箱掉落真源拆到 `chest.xlsx`；在线每只怪死亡掷一次小怪宝箱，最后一波最后一只怪额外掷一次关底宝箱；`offline.xlsx` 不再拥有宝箱概率/类型权重，只控制离线场次、胜率、金币和经验，离线按关卡怪物总数 + 关底最后一击复用 `chest.xlsx` 规则模拟。理由：宝箱应该来自正常刷关体验，离线只是微信小游戏下线后的快速折算，不能变成另一套独立掉落规则；宝箱延迟开出仍可避免离线直接塞爆背包。

- **(2026-07-01) 宝箱开箱先做“可验收最小面板”**：右上奖励卡打开库存，第一版只支持开启首个宝箱，按宝箱类型和来源掉落组生成金币/经验/装备；开箱前先检查背包+仓库总空间，空间不足不消耗宝箱。理由：先把“离线产宝箱→玩家上线看到库存→手动开出奖励”的核心闭环跑通，同时保护玩家资产不因背包满而丢失。开箱倍率暂放 `ChestService`，后续若需要策划调参再迁到配置表。

- **(2026-07-01) 宝箱奖励展示继续放在入口弹层里打磨**：宝箱面板升级为库存列表选择 + 选中预览 + 本次开箱奖励卡片；第一版曾显示金币/经验/装备品质/部位/入库目标，后续规则修正见下一条；空间不足仍沿用“不消耗宝箱”。理由：阶段 3 的目标是把核心开箱反馈做成可验收体验，不先引入独立 UI 框架；等宝箱倍率配置、材料产出和正式视觉确定后，再抽更通用的奖励弹窗组件。
- **(2026-07-01) 修正：宝箱只开装备/材料，并限制库存上限**：宝箱开箱奖励去掉金币/经验，只产装备和材料（打造石/宝石碎片/铭文粉尘），材料进入 `PlayerData.materials`；宝箱库存默认上限 30，在线击杀/关底和离线 claim 都通过 `ChestInventoryModel` 入库，满了后溢出宝箱不保存。理由：金币/经验应来自战斗、离线等常规收益，宝箱定位改为装备成长与材料成长；容量上限避免离线或高频掉落无限堆积，也给后续宝箱管理/扩容系统留下空间。

- **(2026-07-04) 第一章 10 关采用台阶式节奏，卡点在第 4/7/10 关**：难度不做平滑递进，而是三个台阶：裸装可过 1~3 关，第 4 关需穿满 1~3 关掉落，第 7 关需 4~6 段装备成型（合成 T1/T2 补短板），第 10 关纯数值 Boss「屠夫领主」需 7~9 段装备。理由：台阶制造「打→卡→刷→过」循环，同时给掉落/合成/重打三个系统制造存在感；平滑递进会让刷装和合成没有目标感。设计文档见 `docs/superpowers/specs/2026-07-04-chapter1-levels-drop-pacing-design.md`。

- **(2026-07-04) 掉落按台阶分 4 组而非每关一组**：`drop.xlsx` 只配 `c1_early/c1_mid/c1_late/c1_boss` 四组（1~3 / 4~6 / 7~9 / 10），装备等级区间随段位递进且相邻重叠（1-5 / 4-9 / 8-14 / 12-18），Boss 组一次掉 2 件。理由：配置量小、同段内刷任意关收益相同（避免"只刷某一关"的平衡问题）；「掉落等级挂钩 levelIndex」由现有每关 `dropGroup` 指针机制天然实现，不加新代码。同时 `chest.xlsx` 给 `c1_boss` 配了关底必掉宝箱 + `final_boss` 权重组（normal20/boss60/chapter20），章节宝箱第一次有产出点。

- **(2026-07-04) Boss 第一版纯数值，不加技能机制**：`boss_butcher` 只是 enemyTypes 一行配置（大体型/高血/高攻/30% 减免），Boss 波 = boss ×1 + 疾行者护卫。理由：色块占位期先验证数值卡点，技能机制留给后续「技能/Buff/Debuff」专项，避免提前碰 BattleManager 的地盘。

- **(2026-07-04) 开箱掉落组失效按来源关卡兜底**：`openChest` 发现 `chest.sourceDropGroup` 不在掉落表时，回退用 `BattleConfig.levels[sourceLevelIndex].dropGroup`。理由：掉落组这次从 `level_1/level_2` 改名为 `c1_*`，老存档已入库宝箱身上还带着旧组名，没有兜底会直接 throw 崩掉开箱。

- **(2026-07-04) 基准面板校准：dps ×5.5、装备 hp/atk 平铺值同倍放大**：`sim:pacing`（用纯逻辑 BattleManager 头铁模拟）发现旧 dps 面板（hp90/atk28）裸装连第 1 关都过不了——和单只丧尸 1v1 都是五五开，而关卡是群殴。修正：dps 基础 hp 500 / atk 155（≈×5.5），同时 `equip.xlsx` 的 hp/atk 平铺值 ×5（武器 atk 60、头 200/胸 300/腿 225、词条 hp140/atk25），百分比类词条不动。理由：用户选择保留丧尸潮群殴密度（而非压缩关卡怪量）；装备加成是平铺值，基础面板放大后若装备不跟着放大，装备相对价值塌掉、台阶消失。**经验：战斗能力≈战力的平方**（杀得快 × 活得久叠乘），胜负阈值悬崖很陡（×4 全输 → ×5 全赢），所以台阶关强度要按装备档位的平方级差距铺，L4~L10 的怪 hp 覆盖值全部经 sim 逐关校准（13 门槛全达标）并烘焙进种子，改数值后要重跑 `npm run sim:pacing` 回归。

## 表现 / 背景

- **(2026-06-26) 占位美术全用 Graphics 色块**：士兵=方块、怪=圆、子弹=黄点、近战=白线、飘字反馈（伤害/暴击/格挡/闪避）。先调玩法手感，美术后置。
- **(2026-06-26) 可替换的多层视差滚动背景**：独立背景节点（最底层），代码画蓝天/远山/分速度云/地面+滚动纹理。**换真实美术时给该节点加 Sprite 即可，不动战斗代码**。参数在 `BattleConfig.scene`。
- 性能备注：背景的静态渐变目前每帧重画（~48 次 fill），是已知小浪费；接真实贴图后归零，暂不优化。
- **(2026-06-28) 战斗渲染套 `_battleRoot` 容器**：背景/色块/角色 Sprite/飘字/HUD 全挂这一个容器节点，背包面板作其兄弟、打开时 `setSiblingIndex` 置顶。理由：角色 Sprite/飘字是异步或战斗中新建、加到根节点会渲染在面板**上层**（冒出来）；统一进容器 → 战斗里后续新建什么都在面板之下。**以后做任何全屏弹层 UI 都用这个模式**（容器装内容、弹层置顶）。
- 占位 PNG 序列帧改为各帧同色（消除 6fps 亮度脉动导致的"人物闪烁"）；真实美术换帧后才有动画表现。
- **(2026-06-30) 角色动作状态由 `BattleManager` 输出，资源由 `ArtManifest` 绑定**：单位统一维护 `idle/run/attack/death`，移动/实际出手/治疗/死亡只改变逻辑状态；`BattleEntry` 根据状态选择 `char/<职业>/<动作>` 序列帧，缺图仍可回退色块。理由：动作触发属于战斗事实，播放什么资源属于表现层；这样后续替换跑动、普攻、死亡美术只换 PNG 和 Manifest，不把资源路径写进战斗逻辑。
- **(2026-06-30) 启动流程加加载页/开始页门控**：进场先显示 `BootFlow`，等待战斗美术资源和存档都就绪后显示开始页，点击“开始游戏”才激活 `_battleRoot` 并启动战斗。理由：避免资源异步加载期间先露出 Graphics 色块/占位 UI，同时给后续用户提供的加载界面、开始界面留稳定接入点；主界面和战斗仍保持同一 `BattleEntry` 驱动，不新增场景切换复杂度。
