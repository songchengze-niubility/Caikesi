## 项目概述

- **类型**：微信小游戏，Cocos Creator 3.8.8。
- **当前方向**：竖屏整体、**核心战斗横向展开**的多人小队自动战斗（参考「向僵尸开炮」自动战斗 + 永恒蔚蓝星球视角）。
- **小队**：3 人——坦克（近战，前）/ 输出（远程）/ 治疗（后），单排一字阵。
- **当前阶段**：战斗框架 + 关卡系统 + 胜利按配置掉装备 + 结算页/关卡推进 + 装备属性接战斗 + 占位美术。**未做**：技能、Buff、真实美术。
- **原始骨架**：项目最初是「放置挂机刷装备」demo（GameManager/DataService/EventCenter 还在），后转向做战斗。

## 玩法定位决策

- **(2026-06-26) 整体竖屏，但核心战斗横向展开**：小队在左、敌人从右进场向左推，战斗集中在屏幕中部横向带。理由：竖屏放置/RPG 的主流战斗布局，上下留白给 UI。
- **(2026-06-26) 单排一字阵，无碰撞**：双方各排成一条线（y=0），纯靠前后（x）分站位。怪无碰撞、可叠在一起群殴。理由：用户明确要这种「站位分前后、不要分行、可叠」的简单阵型。
- **(2026-06-26) 固定站位 + 自动开火**（非拖动操作）、**固定波次打完即胜**：第一版先验证战斗循环与数值，走位后置。
- **职业分工靠站位**：坦克近战冲到最前当肉墙挡住怪，怪贴脸打坦克；输出远程躲后排集火；治疗奶血量最低的队友。形成「扛—奶—输出」循环。
- **(2026-06-30) 当前阶段切到单人近战 dps 验收**：`battle.xlsx` 增加 `Misc.roster=dps`，当前只生成 dps；tank/healer 的 Stats/Classes/资源仍保留。理由：用户要先不用 3 人小队、只做 1 人 dps；用 roster 覆盖能快速切当前玩法形态，同时不破坏装备/美术/测试里已经存在的多职业基础，后续要回到多人只改表。

## 战斗框架决策

- **(2026-06-26) 逻辑与渲染分离**：`BattleManager` 纯数据逻辑，`BattleEntry`/`Background` 负责画。换美术只动渲染层，玩法数值不受影响——这是「色块占位」策略的基础。
- **(2026-06-26) 减法伤害公式 + 保底**：`伤害 = max(攻-防, 攻×minDamageRate)`。保底防「防≥攻」完全免伤。所有伤害走 `CombatFormula.calcDamage` 唯一入口，便于以后换公式。
- **(2026-06-26) 战斗维度**：暴击（率+伤害加成）、格挡（率+减伤比例）、闪避（完全免伤）、攻速（间隔=基础/攻速）、伤害加成（攻）、伤害减免（防）。结算顺序：闪避→减法→暴击→格挡→加成→减免。
- **(2026-06-26) 暴击伤害用「加成式」(critDmg) 而非倍率**：`暴伤=基础×(1+critDmg)`。理由：装备/Buff 的「+30% 暴伤」可累加，倍率式不好叠。**为掉装备铺路**。
- **(2026-06-26) 统一属性表 `stats`**：所有战斗属性收进一张 `CombatStats` 表，角色和怪共用；单位持 `stats` 引用（实时调参生效）。加新维度只动 `CombatStats`+表+公式。
- **(2026-06-26) 射程 `range` 并入 stats 表**（原在 classes 行为里）。理由：让「+射程」装备词条和其他属性走同一叠算路径。**为掉装备铺路**。
- **近战移动**：坦克冲向最前的怪贴脸再劈，清场退守。**硬保险**：前压位置 ≤「怪线−射程」，不管怎么调参坦克都不会越到怪前面（曾出过「怪跑到坦克后面」的 bug）。
- **怪被秒在天边的修复**：远程射程从无限改有限（dps range 320），并提高怪血量，保证怪能挺进到坦克面前再被集火。

## 关卡系统

- **(2026-06-26) 怪物类型表 `enemyTypes` + 关卡表 `levels` 全配置化**：
  - 怪类型（丧尸/疾行者/重装）各有属性、速度、体型、攻击间隔、颜色。
  - 关卡每关多波，每波可**混合多种怪**（类型+数量+间隔+可选 hp 覆盖）。
  - 打完当前关所有波 = 通关。`startLevel` 选默认进哪关。

- **(2026-06-28) 关卡进度独立成 `ProgressModel`，胜利结算页驱动下一关**：`ProgressModel` 只管当前关、最高解锁和通关后解锁下一关；`BattleEntry` 负责在胜利瞬间发奖励、保存进度并显示结算页（下一关/背包/重打）。理由：关卡进度不是装备存储，也不应埋进 `BattleManager`；`BattleManager` 只判断本局胜负，跨局推进由入口层串联。

- **(2026-06-28) 多系统保存共用 `PlayerDataStore` 缓存**：装备和进度都会写 `PlayerData`，因此抽出 `core/data/PlayerDataStore.ts` 作为共享 load/save 缓存。理由：如果 `InventoryPersistence` 和 `ProgressPersistence` 各自缓存一份 PlayerData，后保存的一方可能用旧快照覆盖另一方字段；共享缓存让本地存档在 MVP 阶段保持简单且不互相踩。

## 表现 / 背景

- **(2026-06-26) 占位美术全用 Graphics 色块**：士兵=方块、怪=圆、子弹=黄点、近战=白线、飘字反馈（伤害/暴击/格挡/闪避）。先调玩法手感，美术后置。
- **(2026-06-26) 可替换的多层视差滚动背景**：独立背景节点（最底层），代码画蓝天/远山/分速度云/地面+滚动纹理。**换真实美术时给该节点加 Sprite 即可，不动战斗代码**。参数在 `BattleConfig.scene`。
- 性能备注：背景的静态渐变目前每帧重画（~48 次 fill），是已知小浪费；接真实贴图后归零，暂不优化。
- **(2026-06-28) 战斗渲染套 `_battleRoot` 容器**：背景/色块/角色 Sprite/飘字/HUD 全挂这一个容器节点，背包面板作其兄弟、打开时 `setSiblingIndex` 置顶。理由：角色 Sprite/飘字是异步或战斗中新建、加到根节点会渲染在面板**上层**（冒出来）；统一进容器 → 战斗里后续新建什么都在面板之下。**以后做任何全屏弹层 UI 都用这个模式**（容器装内容、弹层置顶）。
- 占位 PNG 序列帧改为各帧同色（消除 6fps 亮度脉动导致的"人物闪烁"）；真实美术换帧后才有动画表现。
- **(2026-06-30) 角色动作状态由 `BattleManager` 输出，资源由 `ArtManifest` 绑定**：单位统一维护 `idle/run/attack/death`，移动/实际出手/治疗/死亡只改变逻辑状态；`BattleEntry` 根据状态选择 `char/<职业>/<动作>` 序列帧，缺图仍可回退色块。理由：动作触发属于战斗事实，播放什么资源属于表现层；这样后续替换跑动、普攻、死亡美术只换 PNG 和 Manifest，不把资源路径写进战斗逻辑。
- **(2026-06-30) 启动流程加加载页/开始页门控**：进场先显示 `BootFlow`，等待战斗美术资源和存档都就绪后显示开始页，点击“开始游戏”才激活 `_battleRoot` 并启动战斗。理由：避免资源异步加载期间先露出 Graphics 色块/占位 UI，同时给后续用户提供的加载界面、开始界面留稳定接入点；主界面和战斗仍保持同一 `BattleEntry` 驱动，不新增场景切换复杂度。

- **(2026-06-30) 序列帧抠图默认用站地锚点稳定**：`sprite-keyframe` 新增 `anchor-mode`，默认 `foot` 不是直接跟脚底像素锁横向，而是“身体核心锁 x、脚底落地线锁 y”，保留 `body` 作为身体中心模式。理由：idle 视频看似稳定，但抠白底后透明内容包围盒会被头发、衣摆、呼吸和残留边缘带着漂移；直接拿脚底像素锁 x 会被脚尖/阴影/裙摆形变放大晃动，当前 dps idle 测到旧 foot_x 漂移约 16px，改为身体核心 x 后约 0.5px。游戏角色更需要“横向身体稳、纵向脚站地”，所以 idle/run/站桩攻击优先站地稳定，离地或夸张攻击再切身体中心。
- **(2026-07-01) 序列帧网页预览改为游戏导入帧预览**：`sprite-keyframe` 处理完成后额外生成 `game_keyframes/`，网页动画/深色底/strip/缩略图优先播放这批共同透明边界裁切后的帧；导入 Cocos 时也优先复制同一批 PNG。理由：旧网页预览看的是导入前 512 统一画布帧，导入游戏又裁一次，Cocos 还会受 `.meta` 影响，导致“网页看着对、游戏里不对”；预览和导入必须共用同一套 PNG 才能定位真实抖动问题。

## 装备系统

- **(2026-06-27) 装备存储系统（纯存储层，不影响战斗）**：`assets/scripts/inventory/`——背包/仓库（格子上限）+ 5 装备栏（武器/头盔/胸甲/裤子/鞋子）+ 调试掉落 + 转移 + 穿脱 + DataService 持久化 + Cocos 占位 UI。**关键取舍**：① 范围只到存储层，装备**暂不改战斗数值**（避免一上来就做 effective-stats 层、碰战斗逻辑）；② 装备只有 部位/名字/品质，**无数值、无 equip.xlsx**（占位，YAGNI）；③ Model 纯逻辑不依赖 cc → tsx 单测（11/11）；④ 持久化复用 DataService（`PlayerData.inventory?`）。**实现细节**：`equip` 用「先移除再放回旧装备」，净背包数不增 → 换装免回滚（spec 的"满则回滚"是空条件）。下一版接 effective-stats 让装备影响战斗。

- **(2026-06-27) 改：每角色一套装备栏**：最初设计是「全局一套 5 栏」（按用户"先做一个角色"的话），但 3 人小队共用不合理 → `equipped` 改成 `Record<CharacterId, 部位->装备>`，`equip/unequip` 加角色参数。UI 选「角色切换」（顶部三按钮，选中谁显示谁的 5 栏，穿/脱给当前角色），而非三行同屏（交互简单、不挤屏）。角色 id 复用 tank/dps/healer 但定义在 `EquipDefs`，不耦合战斗配置。

- **(2026-06-28) 装备影响战斗走 effective-stats 胶水层**：新增 `combat/EffectiveStats.ts` 计算「基础 CombatStats + 已穿装备 stats」，`BattleManager` 只多收一个可选 effective map；`InventoryModel` 仍只负责存储/穿脱/序列化，`EquipItem.stats` 来自 `equip.xlsx` → `EquipConfig`。理由：保持存储、配置、战斗公式三者边界清楚；装备加成不绕过 `CombatFormula`，也不把背包模型耦进战斗流程。老存档无 `stats` 的装备按 0 处理；穿/脱装备当前选择“重开战斗刷新属性”，先保证行为可验证，后续若要局内无缝换装再专门做状态迁移。

- **(2026-06-28) 装备 MVP 最低标准：可读详情 + 非固定属性**：固定「部位基础 × 品质倍率」会让同品质同部位装备完全同质，玩家看不到刷装价值；因此 `equip.xlsx` 增加品质 roll 范围、附加词条数量和 `Affixes` 词条池，掉落时每件装备生成实例 `stats`。UI 同步补选中装备详情面板：显示完整属性、品质/部位、与当前角色同部位已穿装备的差值。理由：MVP 至少要能回答“这件装备有什么属性、比我身上这件强弱在哪、为什么两件装备不一样”。

- **(2026-06-28) 装备 UI 交互最低标准：点选看详情、滑动列表、选中后拖拽投放**：背包/仓库列表不再全量铺开，而是可滑动窗口；点击装备负责选中和展示详情，已选中的装备再拖拽到装备栏/背包/仓库完成穿戴、脱下或转移。理由：装备 UI 的 MVP 不能只靠底部按钮，必须支持玩家预期的直接操作；同时“未选中先滑动、选中后拖动”避免列表滚动和拖拽互相打架。为保证拖拽不绕背包造成失败，`InventoryModel` 增加 `equipFromWarehouse` 和 `unequipToWarehouse` 两个直接路径。

- **(2026-06-28) MVP 关卡胜利奖励先用“必掉 1 件装备”打通闭环**：`BattleEntry` 只监听 `BattleManager.phase` 从非 `won` 进入 `won` 的瞬间发奖，优先进背包，背包满则进仓库；不把奖励逻辑塞进 `BattleManager`。理由：先打通“打赢→获得装备→穿上变强→再打”的最小闭环；随后用独立掉落表替换临时随机品质。

- **(2026-06-28) 掉落规则独立成 `drop.xlsx`，关卡只放 `dropGroup` 指针**：`battle.xlsx/Levels` 增加 `dropGroup`，每关决定使用哪套掉落；实际掉几件、品质权重、部位权重由 `tools/config-xlsx/drop.xlsx` 的 DropGroups/QualityWeights/SlotWeights 管理，生成 `DropConfig.rollDropItems(dropGroup)`。理由：品质权重确实要随关卡变化，但不应把所有权重列塞进关卡刷怪表；“关卡引用掉落组 + 掉落表 owns 全部掉落规则”既满足关卡差异化，又避免战斗表膨胀和奖励逻辑散落。

## 美术资源管线
