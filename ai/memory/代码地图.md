# 代码地图

> 模块 → 职责 → 关键文件。**新增/移动模块或改了职责时必须同步更新这里**，否则导航失真。
> 最后更新：2026-06-29。

## 入口

- **场景** `assets/main.scene`：唯一战斗场景，名为 `main`。
- **挂载脚本** `assets/scripts/BattleEntry.ts`：挂在场景的节点上，是当前唯一需要在编辑器挂载的脚本。

## 战斗核心（玩法逻辑）

| 文件 | 职责 | 改这里当… |
|------|------|-----------|
| `combat/BattleManager.ts` | 纯逻辑：布阵、刷怪、移动、攻击、命中、治疗、胜负。不碰渲染 | 改战斗行为/流程 |
| `combat/EffectiveStats.ts` | 装备 → 战斗属性的纯计算层：基础 `CombatStats` + 已穿装备 `stats`，并做概率/生命/攻速钳制 | 改装备如何影响战斗 |
| `combat/CombatFormula.ts` | **唯一伤害结算入口** `calcDamage(att,def)`：闪避→减法→暴击→格挡→加成→减免 | 改伤害公式/加结算维度 |

## 配置（数值由 Excel 管理，策划改 Excel）

> 改战斗/关卡数值：编辑 `tools/config-xlsx/battle.xlsx` → `npm run config` → 生成 `battle.config.generated.ts`；改装备数值：编辑 `tools/config-xlsx/equip.xlsx` → 生成 `equip.config.generated.ts`；改掉落：编辑 `tools/config-xlsx/drop.xlsx` → 生成 `drop.config.generated.ts`。详见 `ai/skills/配置与关卡.md`。

| 文件 | 职责 |
|------|------|
| `config/BattleConfig.ts` | **只剩类型定义** + `SoldierClass` 联合类型；数值对象 `BattleConfig` 引用生成产物 |
| `config/battle.config.generated.ts` | 导表产物（勿手改），结构 = 历史手写版；BattleManager/ConfigPanel 引用它 |
| `config/EquipConfig.ts` | 装备数值类型 + `calcEquipItemStats(slot, quality)`；按品质倍率、roll、附加词条池生成每件装备实例属性 |
| `config/equip.config.generated.ts` | 装备导表产物（勿手改）：品质倍率/roll/附加词条数量 + 部位基础加成 + 词条池 |
| `config/DropConfig.ts` | 掉落数值类型 + `rollDropItems(dropGroup)`；按关卡掉落组掷品质/部位并生成带属性装备 |
| `config/drop.config.generated.ts` | 掉落导表产物（勿手改）：每个掉落组的掉落件数、品质权重、部位权重 |
| `tools/excel-to-config.ts` | 导表器（**多源驱动**，顶部 `SOURCES` 清单）：读 sheet → 校验（缺字段/引用/键一致/跳号报错）→ 生成产物。`npm run config`。加模块=加一张表+一行 source |
| `tools/config-xlsx/battle.xlsx` | **战斗/关卡数值真源**：Stats/EnemyTypes/Classes/Levels/Misc/Scene 六表；`Levels.dropGroup` 决定胜利奖励使用哪套掉落组 |
| `tools/config-xlsx/equip.xlsx` | **装备数值真源**：Qualities/SlotBonuses/Affixes 三表 |
| `tools/config-xlsx/drop.xlsx` | **掉落数值真源**：DropGroups/QualityWeights/SlotWeights 三表；所有掉落件数与权重都在这里 |
| `tools/seed-battle-xlsx.ts` | 从零重建 xlsx 的种子脚本（极少用）；`tools/_e2e-test.ts` 回环自测 |
| `tools/seed-equip-xlsx.ts` | 从零重建 `equip.xlsx` 的种子脚本（极少用） |
| `tools/seed-drop-xlsx.ts` | 从零重建 `drop.xlsx` 的种子脚本（极少用） |

## 存档与数据

| 文件 | 职责 |
|------|------|
| `core/data/DataService.ts` | 数据源接缝，本地/远程可切换；`PlayerData` 目前含 `inventory?`/`progress?` |
| `core/data/PlayerDataStore.ts` | 共享 PlayerData 缓存与保存入口，装备和进度都经它写回，避免多个系统各自 load/save 覆盖彼此字段 |

## 装备系统（`inventory/` 存储/UI + `EffectiveStats` 接战斗）

| 文件 | 职责 |
|------|------|
| `inventory/InventoryModel.ts` | 纯逻辑（**不依赖 cc**，可 tsx 单测）：背包/仓库 + **每角色一套 5 装备栏**(`equipped[角色][部位]`) + 掉落到背包/仓库、`equip(id,角色)`/`equipFromWarehouse`/`unequip`/`unequipToWarehouse`/转移 + 序列化/读档迁移（旧装备缺 `stats` 时自动补属性）。操作返回 `OpResult`，满了/非法即失败不留半成品 |
| `inventory/EquipDefs.ts` | 部位/品质枚举 + 颜色 + 名字占位池 + `createEquipItem(slot, quality)`/`randomItem()`；新掉落从 `EquipConfig` 生成带 roll/附加词条的 `stats`，并提供 UI 属性格式化 |
| `inventory/InventoryConfig.ts` | 小常量：背包/仓库格子上限 |
| `inventory/InventoryView.ts` | Cocos 占位 UI（覆盖层）：**角色切换** + 当前角色 5 栏 + 背包/仓库滑动列表 + 选中装备完整属性/同部位差值对比 + 选中后拖拽投放；唯一 import cc 的装备文件 |
| `inventory/InventoryPersistence.ts` | 持久化胶水：`loadInventory/saveInventory` 经 `DataService`（缓存 PlayerData） |
| `tools/inventory-test.ts` | InventoryModel 单测（tsx），`npm run test:inventory`，覆盖胜利奖励用仓库掉落兜底、直接仓库穿戴、装备脱到仓库、老存档补属性 |
| `tools/effective-stats-test.ts` | 装备属性叠算单测（tsx），`npm run test:effective` |
| `tools/drop-config-test.ts` | 掉落配置单测（tsx），`npm run test:drop`，覆盖关卡 `dropGroup` 指针、完整权重和按权重生成装备 |

> 持久化复用 `core/data/DataService.ts`（`PlayerData.inventory?` 字段）。读档会给老存档缺 `stats` 的装备补属性；直接构造的异常无 `stats` 装备在 `EffectiveStats` 层仍按 0 加成处理。

## 关卡进度（`progression/`）

| 文件 | 职责 |
|------|------|
| `progression/ProgressModel.ts` | 纯逻辑关卡进度：当前关、最高解锁、通关后解锁下一关、继续/重打的边界钳制 |
| `progression/ProgressPersistence.ts` | 关卡进度持久化：把 `ProgressModel` 接到 `PlayerData.progress` |
| `tools/progression-test.ts` | 关卡进度单测（tsx），`npm run test:progression`，覆盖默认值、通关解锁、下一关、末关和老存档钳制 |

> 装备和进度都通过 `core/data/PlayerDataStore.ts` 共用同一份 `PlayerData` 缓存，避免各自保存时用旧快照覆盖对方字段。

## 渲染与调试

| 文件 | 职责 |
|------|------|
| `BattleEntry.ts` | 驱动 BattleManager + Graphics 画占位单位/飘字；建背景节点（`BgFallback` 占位 Graphics 与 `BgSprite` 真实 Sprite 分离）；接美术资源管线；按 941×1672 强制竖屏舞台铺 `StyledUi` 水墨主界面切片（HUD/章节/奖励/技能/底栏）和透明热区，横屏预览时居中加黑边；挂载装备 UI；按 `ProgressModel.currentLevel` 开战；监听通关后读取当前关卡 `dropGroup` 发奖励、保存进度并显示结算页（下一关/背包/重打） |
| `combat/Background.ts` | 背景占位：多层视差滚动；`setUsingSprite(true)` 后清空/禁用占位 Graphics（让位真实背景图） |
| `debug/ConfigPanel.ts` | 游戏内 HTML 调参面板（仅网页预览，临时手感调试，可导出 JSON） |

## 美术资源管线（`art/`，逻辑键 + 缺图回退；详见 `ai/skills/美术资源管线.md`）

> 代码只认逻辑键，缺图回退色块。Codex 流程：丢图到 `assets/resources/art/` + 在 ArtManifest 登记 → 人开编辑器导入生成 `.meta` → 游戏自动用上。

| 文件 | 职责 |
|------|------|
| `art/ArtManifest.ts` | **逻辑键→资源描述**的唯一权威（Codex 维护）+ `entryFiles` 路径助手。不依赖 cc |
| `art/ArtRegistry.ts` | 加载/缓存/回退（泛型 + 注入 loader，不依赖 cc）：`preload/getSprite/getFrames/missingKeys` |
| `art/FrameClock.ts` | 序列帧帧下标纯函数 `frameAt`（不依赖 cc） |
| `art/CocosArtLoader.ts` | cc 胶水：`resources.load` 实现真实 loader → `createArtRegistry()` |
| `art/FrameAnim.ts` | cc 胶水：`FrameAnimPlayer` 代码驱动切帧 |
| `assets/resources/art/` | 资源目录（现为水墨背景 `bg/main`、占位角色序列帧、UI 运行切片等） |
| `assets/resources/art/ui/battle/` | 战斗主界面第一版 raster 运行切片：顶部 HUD、章节/波次、右上奖励卡、3 技能按钮、底部导航；逻辑键登记在 `ArtManifest`，交互热区在 `BattleEntry` |
| `tools/check-art.ts` | `npm run check:art` 校验 ArtManifest 每个文件在磁盘存在；`tools/art-test.ts` 纯逻辑单测 |
| `tools/check-ui-alpha.ts` | `npm run check:ui-alpha` 校验运行 UI PNG 是否带透明边；全不透明 UI 只允许显式例外（完整面板） |

## 规模化约定（代码长大时）

- 新增系统（如 装备/存档/UI/网络）：在对应区新建子目录（如 `equip/`、`ui/`），并在本表加一行 + 必要时加一个 `ai/skills/<领域>.md`。
- `combat/` 文件多了再按子系统拆（如 `combat/spawn/`、`combat/ai/`）。
