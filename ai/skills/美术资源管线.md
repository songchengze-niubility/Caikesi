# 技能 · 美术资源管线

代码用**逻辑键**要资源，缺图自动回退色块。换美术 = 丢图 + 登记，不碰玩法代码。

> GPT 网页生图、Figma 整理、风格总结、UI/美术示意图保存，先走 `ai/skills/UI与美术产出管线.md`，在 `docs/visual/` 保留参考、提示词、导出图和人工审核记录。本文件只处理**用户已经审核通过且确定要进游戏运行**的资源落地。

## 加一份美术（Codex 照做）
0. 确认用户已经明确审核通过并要求进入游戏运行；没有明确通过记录时，停止在 `docs/visual/` 的上游归档，不进入游戏资源目录。
1. 把图按目录约定放进 `assets/resources/art/...`：
   - 静态：`art/<分类>/<名>.png`（如 `art/bg/main.png`）。
   - 序列帧：`art/<分类>/<名>/idle_0.png … idle_{N-1}.png`（编号从 0 连续）。
2. 在 `assets/scripts/art/ArtManifest.ts` 登记逻辑键：
   - 静态：`'bg/main': { type: 'sprite', path: 'art/bg/main' }`
   - 序列帧：`'char/tank/idle': { type: 'frames', dir: 'art/char/tank/idle', frames: 4, fps: 6, loop: true }`
3. `npm run check:art` 确认文件齐全（缺文件会列出来）。
4. `git commit`（图 + ArtManifest）。

## 序列帧规格红线（显存，2026-07-04 定）

角色/怪物动画正式方案就是序列帧（不引入 Spine，理由见 `设计日志.md`）。显存靠三条红线控制，新增/替换序列帧前逐条对照：

1. **帧高 ≤ 游戏内显示高度 ×2**。角色屏显约 120px → 帧高上限 240px。`npm run sprite:keyframes:ui` 里「游戏帧高上限」默认 240：`game_keyframes` 超高会自动等比缩小，导入即合规；离地大招等特殊动作按实际显示高度另算，但同样 ×2 封顶。
2. **每动作 ≤10 帧**（显存预算本体）；fps 本身几乎免费（只是轮播速度），**上限 18fps**——约束因此转到循环时长：`时长上限 = 10帧 ÷ fps`（18fps→0.56s，12fps→0.83s）。取舍标准（2026-07-05 用户定）：**宁短勿卡**——循环短、重复感强可接受，fps 低、逐格卡顿不可接受；剪辑时按目标 fps 倒推循环单元长度。
3. **微信小游戏构建开压缩纹理**（人工一步，只影响构建产物、不改源 PNG）：
   - Cocos Creator → 项目设置 → 压缩纹理：新建预设（微信小游戏平台：ASTC 6x6 优先，ETC2 兜底）。
   - 资源管理器多选 `resources/art/` 下的角色/怪物贴图 → 属性检查器套用该预设（新增序列帧后记得补选）。
   - 构建微信小游戏包验证产物生成 `.astc`；显存约降到 RGBA8888 的 1/9~1/16。

> 换算参考：RGBA8888 显存 = 宽×高×4 字节/帧。240px 高一帧约 150KB，10 帧×4 动作 ≈ 6MB，ASTC 后 <1MB/角色。

## UI 运行资源额外门槛

UI 示意图来自一张扁平截图/GPT 成图时，不能把矩形裁图直接转进游戏运行资源。进 `assets/resources/art/ui/...` 前必须已经完成：

1. 组件拆件：每个 UI 组件单独 PNG，组件外区域透明；`raw` 裁图只留在 `docs/visual/` 当参考。
2. HTML 预览：生成 `preview.html`，每个组件至少叠在棋盘格和游戏近似背景色上检查脏底。
3. 用户审核：明确记录通过，且列出允许晋级到游戏资源的文件。
4. 透明校验：复制到运行资源目录后运行 `npm run check:ui-alpha`；全不透明 PNG 只有“组件本身就是完整面板”时才允许，并在脚本例外里写理由。
5. 视觉确认后再登记 ArtManifest；否则退回上游重抠、手工 mask、索要源分层或重绘。

## 人工一步（必须）
- 开一次 Cocos 编辑器，等它自动导入新图（生成 `.meta`），把 `.meta` 一起提交。
  Cocos 规定资源须经编辑器导入获得 uuid，此步不可省。
- **序列帧多一步**：Cocos 生成的默认 `.meta` 是 `trimType:auto`，游戏 `SizeMode.CUSTOM` 下会拉伸抖动。
  Cocos 导入后在 sprite-keyframe 页面对该目录点「修补 .meta」（或重新覆盖导入）。
  `npm run check:art` 会校验全部序列帧 meta（trimType=custom、尺寸=PNG、同动作同尺寸），漏修直接报红。

## 序列帧身体基准（跨动作一致性）

frames 条目带可选 `bodyH`/`anchorX`/`footY`（0-1 归一化，sprite-keyframe 导入时自动计算，含义见
`ArtManifest.ts` 顶部注释）。游戏侧按 bodyH 把身体缩放到职业目标高、按 anchorX/footY 对齐脚点，
避免各动作独立裁切导致的切动作身体忽大忽小/横移。不填 = 整帧缩放旧行为；指标不准可手改微调。

## 不变量
- **路径只在 ArtManifest**；代码（渲染层）只用逻辑键，缺图回退色块、不报错。
- 资源放 `assets/resources/art/` 下才能按路径加载。
- 加载集中在进场 `preload`，不在战斗每帧（性能约束）。
- `ArtManifest`/`ArtRegistry`/`FrameClock` 不依赖 cc（可单测）；cc 只在 `CocosArtLoader`/`FrameAnim`/渲染接入。
- 路径权威=ArtManifest（被 check-art 校验）；操作权威=本文件。
- UI 组件权威=透明 PNG + HTML 预览 + alpha 校验；扁平原图只作 reference。
- 审核权威=用户明确通过记录；没有用户审核通过记录，不得进入 `assets/resources/art/`。

## 自检
- `npm run test:art`（纯逻辑）+ `npm run check:art`（文件齐全）。
- UI 资源另跑 `npm run check:ui-alpha`（防止示意图背景被裁进组件）。
- 游戏内"缺图"浮层列出还在用色块的键。
