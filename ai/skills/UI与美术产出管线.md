# 技能 · UI 与美术产出管线

这条管线负责**上游视觉生产**：参考图、提示词、Figma 画板、UI 示意图、风格沉淀。类 VberAI 的本地 UI 拆分工作台已按用户要求删除；如需从扁平 UI 图里临时取素材，可用精简 `tools/ui-slicer/` 手动框选、轻量去底和导出 PNG，再拿去 Figma 加工。后续新导出的 UI / 美术图片先归档到 `docs/visual/` 并保留人工审核记录。只有用户明确审核通过并确定要进游戏运行时，才继续走 `ai/skills/美术资源管线.md` 放入 `assets/resources/art/` 并登记 ArtManifest。

角色参考图要用 Codex 生产四状态固定 8 帧时，具体生成、去底、锚点、审核与返工走 `ai/skills/序列帧角色制作.md`；本文件只保留上游参考、风格与审核记录的通用约定。

## 目标

- 用户用 GPT 网页版生成图和提示词，把图/提示词发给 Codex。
- Codex 在 Figma 中整理参考、搭 UI 示意、标注取舍。
- Codex 更新 `ai/memory/视觉风格.md`，沉淀当前风格、禁忌、可复用提示词积木。
- Codex 把新导出的图片、提示词、Figma 导出和人工审核记录保存到 `docs/visual/`。
- 用户审核通过后，确认要进游戏运行的资源才进入 `assets/resources/art/`。
- 下一轮需要更多图时，Codex 根据风格记忆给用户一段可直接丢给 GPT 网页版的提示词。

## 输入包（用户发图时先补齐）

最小输入：
- 原图或截图。
- GPT 网页版使用过的提示词。
- 用途：`角色` / `怪物` / `背景` / `UI` / `图标` / `特效` / `风格探索`。

最好再问清：
- 要保留什么：轮廓、配色、材质、构图、情绪、UI 结构。
- 要避开什么：太写实、太暗、太花、细节过密、不可读、偏离竖屏战斗。
- 目标画幅：竖屏 UI 通常 `9:16`；横向战斗背景可 `16:9` 或按 Cocos 视口裁切；独立资产优先透明背景。

## Figma 整理

如果本环境可调用 Figma 工具，先按工具要求加载对应 Figma skill，再操作文件。

建议使用一个长期文件：`Cgame Visual Pipeline`。页面约定：
- `00 Inbox`：用户发来的原图、截图、原始提示词，按日期入箱。
- `01 Style Board`：风格板，提炼色彩、形状语言、材质、光影、UI 质感。
- `02 UI Mockups`：竖屏界面示意，优先做战斗主界面、背包/装备、结算、关卡选择。
- `03 Game Art Sheets`：角色、怪物、背景、特效、图标等资产表。
- `04 Export Queue`：准备导出的精选帧。
- `99 Archive`：被淘汰但可能回看的探索。

命名格式：
- Figma frame：`YYYY-MM-DD_用途_主题_vNN`，如 `2026-06-28_ui_battle-main_v01`。
- 参考图标注：`来源 / 原提示词 / 用途 / 保留点 / 淘汰点`。
- UI 示意图必须用竖屏第一视口判断可读性，战斗画面保持“竖屏整体、核心战斗横向展开”。

## UI 拆件与 HTML 预览

用户给 UI 示意图时，默认目标是：**先整理可追溯的参考、取舍和人工审核记录**；如果要进入游戏，再按 `美术资源管线` 准备干净的组件 PNG。不要把整张图或矩形裁图直接当运行资源。

本地辅助工具：
- `npm run ui:slicer` 打开 `tools/ui-slicer/index.html`，用于手动框选、轻量去底、吸管取底色、遮罩画笔修边，并把 PNG 下载/复制到 Figma。
- 推荐素材落点：`tools/ui-slicer/storage/ui/full-reference/` 放完整参考图，`backgrounds/` 放 UI 背景图，`components/` 放导出的 UI 组件。
- `ui-slicer` 的产物默认仍是上游中间素材：可进 Figma 和 `docs/visual/`，不能因为带透明背景就直接进 `assets/resources/art/`。

必做：
- 在 `docs/visual/` 保存本轮原图、参考图、提示词和审核说明。
- 如有人工拆件，组件另存为独立 PNG，并记录原图坐标/尺寸和命名；组件外区域必须透明。
- 保留预览或截图记录：每个组件至少同时叠在棋盘格和游戏近似背景色上检查脏底。
- 生成 alpha/验收记录：透明比例、哪些组件允许不透明、允许理由。
- 等用户记录明确通过后，才允许考虑进入游戏。

硬规则：
- `raw rectangular crop` 只能作为 reference，不能直接进入 `assets/resources/art/`。
- 没有用户明确审核通过，任何上游视觉图片都不能复制到 `assets/resources/art/`。
- 文字、数值、波次等动态内容优先在 Cocos/HTML 中重建为真实 Label；只有确认永远不变的装饰文字才可烘焙进 PNG。
- 全不透明 UI PNG 默认视为可疑；只有“组件本身就是完整矩形面板”（如奖励卡、整条底栏、纸质标题牌）才允许不透明，并且要在记录或校验脚本里写理由。
- 从扁平截图自动抠不干净时，不要静默入库；改为标记“需要源 PSD/Figma 分层/手工 mask/重绘组件”。

## 风格总结

每次 Figma 整理结束，更新 `ai/memory/视觉风格.md`。只写当前仍有效的风格事实：
- 核心风格锚点：世界观、情绪、明暗、夸张程度。
- 色彩与材质：主色、强调色、背景/角色/UI 的对比关系。
- 角色/怪物规则：轮廓、比例、可读性、阵营差异。
- UI 规则：信息密度、按钮/面板质感、字体气质、战斗 HUD 取舍。
- 禁忌：不要再生成或设计的方向。
- 提示词积木：可复用的正向 prompt、反向 prompt、画幅/透明背景约束。

不要把每张图的完整历史堆进视觉风格；历史放 `docs/visual/prompts/`，风格记忆只保留“下一轮要继承的结论”。

## 仓库保存

`docs/visual/` 是视觉追溯库：
- `references/`：用户发来的原始图/截图，只存精选或有决策价值的参考。
- `prompts/`：每轮提示词记录，含原 prompt、Codex 改写 prompt、结果评价；新记录按 `docs/visual/prompts/TEMPLATE.md` 填。
- `exports/ui/`：Figma 导出的 UI 示意图。
- `exports/art/`：Figma 导出的美术资产示意或风格稿。
- `exports/style/`：风格板、色板、组件质感板。

命名格式：
- `YYYY-MM-DD_<用途>_<主题>_vNN.png`
- `YYYY-MM-DD_<用途>_<主题>.md`

只有用户审核通过且要进游戏运行的成品图，才复制到 `assets/resources/art/`，然后按 `美术资源管线` 登记逻辑键、跑 `npm run check:art`。

## 回流给 GPT 网页版

给用户下一轮生图 prompt 时，用四段式：

```text
[Cgame 风格前缀]
[本次对象/界面要求]
[构图、画幅、透明背景、可读性要求]
[负向约束]
```

常用约束：
- 竖屏 UI：强调 360px 宽度下可读、按钮清晰、不要遮挡战斗核心区域。
- 游戏资产：干净轮廓、少小碎细节、可在小尺寸识别。
- 背景：留出战斗横向空间，前景不要抢角色剪影。
- 图标：透明背景、正交或轻微透视、一眼识别部位/功能。

## 验收清单

- Figma 中有来源图、原 prompt、取舍标注。
- 新导出图片、提示词、Figma 导出和审核记录已进入 `docs/visual/`。
- `ai/memory/视觉风格.md` 已更新当前结论。
- 用户审核通过后，精选导出进入 `docs/visual/exports/...`，提示词进入 `docs/visual/prompts/...`。
- 下一轮 GPT prompt 已给用户，且继承当前风格而不是从零散想。
- 若图要进游戏，必须有用户审核通过记录，并转入 `美术资源管线`；不把 Figma 草稿、上游参考图或 raw crop 直接当运行资源。
