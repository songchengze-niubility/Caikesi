# 技能 · 开发收尾与交接

每段开发告一段落，按顺序做，给下一个对话留干净的接力棒。

## 1. 更新记忆
- `ai/memory/项目状态.md`
  - 刷新进度 / 方向 / 待办 / **未合并分支状态**；
  - **更新顶部「最近进展（给新对话）」**：3~6 行说清"刚做完什么、下一步、有什么坑"。这是新对话的第一落点。
- `ai/memory/设计日志.md`：本次有方向性决策 → 追加（日期 + 理由）。
- `ai/memory/代码地图.md`：新增/移动模块或改了职责 → 同步。
- **新增 skill 时**：检查有没有**重叠的旧 skill**，更新它/加交叉链接（别只新增不打通）；并在 README 技能索引加一行。

## 1.5 反思：新对话能否完整继承？（每次必做）
顺着入口链 `CLAUDE.md → README → 项目状态 → 代码地图 → 设计日志 → 相关 skill` 自问：**现在开一个全新聊天，它能完整继承当前知识框架吗？** 逐项查并补缺口：
- 代码地图有没有新模块区？新 skill 和重叠旧 skill 打通了吗？
- 项目状态的进展/待办/未合并分支是否反映真相？方向性决策入设计日志了吗？
- 有没有"只在某分支、master 看不到"的知识？→ 在项目状态留指针。
> 标准是「一个零上下文的新对话，靠入口链就能在正确前提上接着干」。

## 2. 防膨胀（三份记忆都要控量；只增不减是病根）
**只增不减的文档迟早撑大「每次必读的固定小集合」，抬高每次会话成本。给它们都设归档/压缩线：**
- **设计日志.md**：超约 200 行或某里程碑完结 → 旧条目**剪切**到 `ai/memory/归档/<年份>-<主题>.md`，主日志留一行指针；只保留"近期 + 仍影响当前开发"的决策。
- **项目状态.md**：「已完成」只留**当前阶段**的，更早的里程碑压成一行（或剪到归档）；「最近进展」永远只 3~6 行（滚动窗口，不堆历史）。
- **代码地图.md**：保持**导航级**（一模块一行职责，不写实现）；某区行数多了按域拆文件（如 `代码地图-战斗.md`/`-美术.md`），主图留指针。

> 原则：记忆是"当前能接着干所需的最小集"，不是变更史——史看 git/设计日志归档。

## 3. 交接给新对话
新对话的接力路径已经由 `CLAUDE.md` 固化：
`CLAUDE.md → README → 项目状态.md(顶部最近进展) + 代码地图.md`。
所以"交接"= 把第 1 步那几个文件更新好即可，**不需要额外写长篇交接文**。

## 4. 性能自检（只在改了热路径时做）
触发条件：本次改了 `update`/`tick`/渲染/寻敌/碰撞/特效/UI 刷新，或铺大了怪/子弹/特效数量。
做法（对照 `ai/skills/性能约束.md` 的「开发收尾检查」）：
- 微信开发者工具跑一次，看 **DrawCall / 帧率 / 内存** 三个数。
- 与 `项目状态.md` 底部「性能基线」对比：明显变差就当回归，先查再提交。
- 基线有意义的变化（如 DrawCall 跳档、新增系统后内存上一台阶）→ 更新「性能基线」那一行。
- 没碰热路径就跳过本步，别为做而做。

## 5. 提交
- git 提交本次改动（中文 message，结尾 `Co-Authored-By: Claude ...`）。
- 只在用户要求时 push。

> 一句话：收尾 = 让「项目状态/设计日志/代码地图」反映真相 + 归档膨胀部分 +（动了热路径才）性能自检 + 提交。
