# 技能 · 性能约束

目标：在项目还没铺量时先立规矩。以后 AI 写代码要默认把微信小游戏真机性能当成约束，而不是等卡了再补救。

## 总原则

- **先控热路径**：`update` / `tick` / 渲染 / 碰撞 / 寻敌 / 动画 / UI 刷新是高风险区，写之前先想规模上限。
- **先数据，后表现**：战斗逻辑保持纯数据；渲染、动画、音效、飘字、特效只消费状态，不反向污染逻辑。
- **先池化，再铺量**：会反复出现/消失的对象（怪、子弹、飘字、伤害特效、音效节点）必须预留对象池方案。
- **先真机，再乐观**：桌面预览流畅不代表微信真机流畅；关键节点必须用微信开发者工具和低端机/性能面板验证。
- **红线可量化**：战斗帧 DrawCall 控制在两位数；目标 60fps，菜单/非战斗可 `game.frameRate` 降频；真机帧率掉到 50 以下、或内存逼近上限就当性能回归处理。

## 每帧代码硬约束

- 避免在每帧里制造短命对象：少用 `new`、对象字面量、数组字面量、闭包、`filter/map/reduce`、字符串拼接。
- 每帧清理数组优先原地复用：用 `length = 0`、倒序删除、swap-remove，少用 `arr = []` / `arr.filter(...)`。
- 距离判断优先用平方距离：热路径里少用 `Math.sqrt` / `Math.hypot`，除非确实需要真实距离。
- 状态文本不要每帧刷新：Label/UI 只在值变化时更新，或低频节流（例如 0.1~0.2 秒一次）。
- 不在每帧查找节点/组件：禁止热路径里 `find`、反复 `getComponent`、按名字扫节点树；初始化时缓存引用。

## 微信平台硬约束（区别于通用游戏，这几条是平台规则不是建议）

- **内存/显存优先于 CPU**：低端机/iOS 内存吃紧会被系统直接杀进程（是闪退不是卡）。大图避免 RGBA8888 全尺寸常驻，不可见关卡/场景资源及时 `releaseAsset`；显存比帧率更容易让低端机崩。
- **包体与分包**：微信主包有硬上限（约 4MB），总包亦有上限。主包只放首屏战斗必需，美术/关卡/音频走**分包加载**；新增大资源前先决定它进主包还是分包。
- **引擎模块裁剪**：项目设置里剔除用不到的引擎模块（3D、物理3D、物理2D 等），减小代码包体与启动开销。
- **慎用同步微信 API**：禁止在热路径或高频处调用同步 `wx.*`（如 `setStorageSync`/`getSystemInfoSync`），会阻塞主线程；系统信息初始化时缓存一次复用，存档用异步或合批落盘。

## 战斗与规模约束

- 写新逻辑时必须估算复杂度：避免无意识写出 `单位数 × 怪物数 × 子弹数` 的循环。
- 碰撞/命中优先纯数学判定（距离平方、AABB），非必要不开物理引擎；当前「无碰撞群殴」是有意为之，保持。
- 寻敌、碰撞、范围检测先按当前规模写简单版，但要留下升级点：最近目标缓存、前排列表、空间分桶、锁定目标。
- 子弹/怪物/特效增加数量前，先加上限：最大存活子弹数、最大飘字数、最大同时特效数。
- 随机、伤害、命中等结算集中在逻辑层，渲染层不重新做玩法判断。
- 配置允许策划调大数量时，要考虑性能护栏：数量上限、间隔下限、极端值兜底。

## 渲染与美术约束

- `Graphics.clear()` 后全量重画只能作为占位/调试方案；真实美术阶段优先 Sprite/Animation/Spine/粒子等可管理节点。
- 静态背景不要每帧重画：拆成静态层（画一次或贴图）+ 动态层（滚动/动画）。
- 跳动数字（伤害飘字）不要用默认 Label：每次改文字会重建纹理。用 `Label.cacheMode = CHAR` 或 BMFont 位图字体。接美术后这是飘字第一个要换的点。
- 真实美术必须考虑合图：同类 UI/角色/特效尽量进 SpriteAtlas，减少纹理切换和 draw call。
- 控制节点数量：大量单位、血条、飘字、特效都要池化；离屏/不可见对象要停动画、停粒子、停更新。
- 透明叠加、粒子、发光、遮罩、RichText、动态图集切换是微信小游戏高风险点，新增时要说明用途和上限。

## UI 与调试工具约束

- 调试面板、日志、监控浮层不能进正式包；至少要有环境开关或构建期排除路径。
- `console.log` 不进热路径；上线前清掉高频日志。
- UI 列表要做复用/分页/虚拟列表；不要一次创建大量 item。
- 红点、战力、金币、血量等数值 UI 走事件或脏标记更新，不每帧扫全局状态。

## 资源与加载约束

- 大图、音频、Prefab、Spine 等资源按场景/系统分批加载，不在战斗中途同步制造卡顿。
- 常用战斗资源进入战斗前预加载；不用的关卡/美术及时释放或交给资源管理策略。
- 音效要限制并发，重复短音效要复用 AudioSource 或统一音效管理。
- 配置数据增长后，读取和转换只在加载/进关时做，不在战斗帧里解析 JSON/表格。

## AI 写代码前自检

涉及性能敏感区域时，先在回复或代码注释外部说明：
- 这个逻辑预计每帧跑多少次？
- 数据规模变大后复杂度是多少？
- 有没有每帧分配？
- 需要对象池/缓存/节流吗？
- 微信真机验证点是什么？

## 开发收尾检查

每段开发结束前，若改到热路径，至少检查：
- 是否新增每帧 `new` / `filter` / `map` / `reduce` / 字符串拼接？
- 是否新增未池化的节点创建/销毁？
- 是否新增全量扫描或嵌套循环？
- 是否新增每帧 Label/UI 刷新？
- 是否需要更新 `ai/memory/项目状态.md` 的性能风险/待办？
