# 技能 · 序列帧角色制作

目标：把一张角色参考图变成可审核、可返工、最终可进 Cocos 的四状态序列帧；审美判断留给用户，机械处理与量化校验交给工具。

## 固定交付契约

- 状态：`idle / run / attack / death`。
- 帧数：每状态**恰好 8 帧**，编号从 0 连续到 7。
- 方向：纯正侧视、始终朝屏幕右侧；代码当前不做水平翻转。
- 格式：透明 RGBA PNG，同动作全部同画布，默认 `320×240`，帧高不得超过 240px。
- 资源边界：用户明确审核前只放 `docs/visual/staging/characters/`，不得进入 `assets/resources/art/` 或改 `ArtManifest`。
- 完整角色默认顺序：`idle → run → attack → death`，逐动作审核，不一次押注 32 帧。用户明确只做某一个动作时，可以跳过顺序，但必须已有审核通过的身份锚点。

## 最重要的返工边界

先判断错误属于哪一层，再决定修法：

| 问题 | 处理 |
|---|---|
| 白边、绿边、透明残渣、画布尺寸、整帧 X/Y 漂移 | 本地后处理 |
| 格数错误、角色/武器越格、整体视角错误 | 整动作重新生成 |
| 头型、眼线、颈部、服装、肢体、武器结构漂移 | **整动作重新生成** |
| 动作节奏不连贯、循环接缝错误 | 重写八拍后整动作重新生成 |
| 单个游离噪点 | 可局部清理 |

**禁止用某一帧的头、手、武器等结构件硬覆盖其他帧。**这种做法可能让像素指标变好，却会破坏头颈、发根、关节、光影和笔触的整体关系。结构漂移是生成问题，必须回生成源头解决。

## 1. 建立角色锁定包

1. 把用户原图保存为：
   `docs/visual/staging/characters/YYYY-MM-DD_<角色>-vNN/reference.png`。
2. 明确输入角色：`Image 1 = identity/style anchor`，不是要直接修改的运行资源。
3. 记录不可变项：
   - 头身比、头型、眼睛/面部表达；
   - 发型、发根和发饰连接点；
   - 服装层次、腰带、鞋；
   - 武器形状、长度、握持手；
   - 朝向、色板、水墨笔触。
4. 原图若不是纯侧视，先生成并审核一张标准侧视立绘，再把它冻结为后续唯一身份锚点。
5. 原图朝左但项目要求朝右时，先生成朝右身份锚点并审核；存在不对称服装、惯用手或单侧武器时禁止直接镜像后默认为同一设计。

## 2. 先写八拍，再生图

每个动作先定义 0~7 帧语义。循环动作的第 7 帧应通向第 0 帧，但不能直接复制第 0 帧，否则接缝会停顿。

| 动作 | 播放 | 默认 fps | 八拍重点 | 对齐模式 |
|---|---|---:|---|---|
| idle | 循环 | 10 | 刚性主体稳定，只做连续次级摆动 | `sideview` |
| run | 循环 | 18 | 完整左右步态；原地跑；禁止 `pingpong` 伪造循环 | `sideview` |
| attack | 单次 | 18 | 预备→蓄力→命中→后摇；命中建议第 4/5 帧 | 站地用 `sideview`，离地用 `body` |
| death | 单次 | 10 | 受击→失衡→倒地；第 7 帧可长期停留 | 倒地后用 `body` 或保留原位移 |

默认 fps 先用于首轮预览；用户按手感调整时只改播放速度，不改变固定 8 帧契约，且不得超过 18fps。

注意：当前攻击表现状态约 0.32 秒；8 帧在 18fps 下完整播放需要至少 `8÷18≈0.445s`。生成 attack 可以先做，但入库前必须延长表现窗口或把动画播放与逻辑状态解耦，不能用约 25fps 绕过项目 18fps 红线。

## 3. 用 image generation 整动作生成

先加载系统 `imagegen` skill，默认使用内置 image generation。每动作单独调用一次；不要用多个独立调用分别画 8 帧，否则身份漂移更严重。

统一提示词骨架：

```text
Use case: identity-preserve
Asset type: production 2D game sprite sheet, exactly eight frames for one <ACTION> animation
Input images: Image 1 is the sole identity, anatomy, costume, weapon and ink-style anchor
Primary request: regenerate the complete eight-frame action from scratch as one coherent animation; do not patch or collage existing frames

Identity invariants: preserve the exact head shape, eye, hair root, proportions, costume construction, weapon design and ink-wash style from Image 1.
View: strict orthographic pure side view, always facing screen-right, fixed camera and scale.
Layout: exactly 4 equal columns by 2 equal rows, left-to-right then top-to-bottom; one complete character per cell; no grid lines, borders, numbers or text.
Scene/backdrop: perfectly flat uniform solid #00ff00 chroma-key background.
Constraints: full body and full weapon inside each logical cell; no shadow, floor, gradient, reflection, motion blur, VFX, watermark or text; do not use #00ff00 in the character.
Avoid: identity drift, random head redraw, neck stretch, costume/weapon mutation, 3/4 view, mirroring, extra/missing limbs, brightness flicker and cell crossing.
```

### idle 专用约束

Idle 优先采用 limited animation，而不是让生成器重画八次整个人：

```text
Build all eight frames from one shared rigid master drawing.
Keep the round head, eye stroke, hairline, hair tie, neck, torso, arms, hands, sword, legs, boots, body center and ground contact visually fixed.
Only the loose ponytail strands after the hair tie, waist sash tails and outer torn robe hem may change through eight smooth wind-sway phases.
Their attachment points remain fixed. The head and rigid body must not wobble, resize, rotate, bob or morph.
```

Run/attack/death 允许有设计意图的头身运动，但轮廓变化必须连续、能解释，不能是随机换头或换比例。

## 4. 保存版本与提示词

每动作使用独立版本目录：

```text
docs/visual/staging/characters/YYYY-MM-DD_<角色>-vNN/
  reference.png
  <action>-v01/
    <action>_sheet_chroma.png
    <action>_sheet_rgba.png
    raw_frames/
    processed/
    final_frames/
    <action>_preview_checker.gif
    <action>_preview_dark.gif
    <action>_contact_rgba.png
    head_diagnostic.png       # idle 必做；其他动作按需
    alpha_qc.json
    prompt.md
```

保留判退版本，不覆盖历史；在 `prompt.md` 标明判退原因。只有当前候选目录使用 `final_frames/`。

## 5. 去底：先键色，再尊重 alpha

1. 生成时要求完全平坦的 `#00ff00` 背景。
2. 用系统脚本转换：

```powershell
py -3 "$env:USERPROFILE\.codex\skills\.system\imagegen\scripts\remove_chroma_key.py" `
  --input <action>_sheet_chroma.png `
  --out <action>_sheet_rgba.png `
  --auto-key border `
  --soft-matte `
  --transparent-threshold 12 `
  --opaque-threshold 220 `
  --despill
```

3. 后续处理已去底 PNG 时使用 `--background-mode alpha`。不要再次按白底或边缘 RGB 去底：透明像素的隐藏 RGB 可能是黑色，二次 edge key 会把黑衣主体一并删掉。
4. 出现细绿边先重跑去底或尝试 `--edge-contract 1`；不要用大范围腐蚀吃掉水墨飞白。

## 6. 切 4×2 动作表

- 读取顺序固定为左到右、上到下。
- 不要默认按等宽边界硬切。长剑、马尾、披风可能越过数学分格线，硬切会断剑或断发。
- 优先给每格保留横向重叠，再按该格中心选择主体连通域。
- 只有武器、发丝、衣摆全部与身体连通时才能保留“最大连通域”；若存在脱手武器、飞散特效或断笔飞白，改用人工 mask 或重生，不能静默删除。
- 切完必须恰好得到 8 张，且每张角色和武器完整、无邻格残片。

对角色/长武器全部连通的动作表，使用确定性入口：

```powershell
py -3 tools/sprite-pipeline/split_ai_sheet.py `
  --sheet <action>_sheet_rgba.png `
  --out <raw_frames> `
  --columns 4 `
  --rows 2 `
  --overlap-x 100 `
  --largest-component-only
```

脚本会强制 8 帧、透明四角、行优先顺序、主体不触边，并校验选中的身体核仍落在原逻辑格内。`overlap-x` 按长剑/马尾越界量调整；脚本报触边时不得忽略。

## 7. 统一画布与锚点

对已去底的 8 帧运行：

```powershell
py -3 tools/sprite-keyframe-tool.py `
  --frames-dir <raw_frames> `
  --select-indices 0,1,2,3,4,5,6,7 `
  --out <processed> `
  --background-mode alpha `
  --alpha-threshold 12 `
  --frame-size 320x240 `
  --padding 4 `
  --anchor-mode sideview `
  --preview-columns 4
```

- `sideview` 已经锁定“身体主干列密度 X + 地线 Y”；不要再对同一批帧叠加旧的视觉重心平移。
- 只允许整体平移和统一缩放，不允许为追指标拉伸、局部形变或硬换头。
- run 的脚交替离地属于动作，不要冻结脚底像素；锁世界根节点和地线即可。
- death 横躺后若 `sideview` 失真，切换 `body`，保留倒地动作的真实位移。

确认 `processed/keyframes/` 后再晋级候选帧，按顺序改名复制：

```text
processed/keyframes/key_000.png → final_frames/<action>_0.png
...
processed/keyframes/key_007.png → final_frames/<action>_7.png
```

不要把 `processed/` 的调试预览当作运行资源，也不要未经审核直接调用导入接口。

## 8. 自动质检

所有动作：

- 8 张、连续编号、同尺寸、RGBA；
- 四角 alpha 为 0；无白底、伪棋盘格、明显绿边；
- 帧高 ≤240px；
- 身体主干/脚点符合动作语义；
- 头型、服装、武器没有随机结构变化；
- 棋盘格、深色底和实际游戏背景都能读清。

Idle 追加门槛（`320×240` 成品帧）：

- 头部暗色核心质心 X/Y 各自范围建议 ≤1px；
- 身体主干 X 范围 ≤1px；
- 脚点 Y 范围 ≤1px，脚点 X 范围建议 ≤2px；
- 头部逐帧放大对比不得出现换头、眼线跳变、颈部伸缩；
- 7→0 接缝变化不得明显大于普通相邻帧。

Run 追加门槛：

- 身体世界根节点 X 范围建议 ≤2px，地线锚点 Y 范围 ≤1px；
- 头部可以随步态形成平滑周期轨迹，但头型/眼线/尺寸不得随机变化，头部宽高范围建议各 ≤2px；
- 左右脚必须完成一次对称交换，任何连续两帧不得突然换腿或换剑结构；
- 以默认 18fps 检查 7→0 接缝和小尺寸可读性。

数值过线但肉眼仍晃，按肉眼判负；数值只是报警器，不是审美结论。

## 9. 给用户看的审核包

每轮完成后直接提供可点击路径，不要只说“已生成”：

- 棋盘格动画 GIF；
- 深色底动画 GIF；
- 4×2 八帧总览；
- idle 的头部逐帧放大对比；
- `final_frames/` 目录；
- 简短说明本轮是“整组重生”还是“仅机械后处理”。

用户判退时，把原因写进该版本 `prompt.md`，新建 `v02/v03`，不要覆盖旧证据。

同一结构问题连续两次整动作重生仍失败时，先收紧动作幅度、简化允许运动部位或重做身份锚点；第三次仍失败则暂停批量生成，让用户选择新锚点、改动作设计或更换生产方式，禁止无限抽卡式重试。

## 10. 审核通过后再入 Cocos

1. 复制到：
   `assets/resources/art/char/<角色>/<action>/<action>_0.png … <action>_7.png`。
2. 在 `ArtManifest.ts` 登记 `frames: 8`：
   - idle/run：`loop: true`；
   - attack/death：`loop: false`。
3. 写入工具计算的 `bodyH / anchorX / footY`。
4. 打开 Cocos 生成 `.meta`，再修补为 `trimType: custom`、full-frame、同动作同尺寸。
5. 运行：

```powershell
npm run test:art
npm run check:art
npm run verify
```

6. 用动作预览面板和实际战斗检查切状态是否跳大、横移、上下跳线；微信构建继续使用 ASTC 6×6 / ETC2 压缩预设。
